## Objective

Let the user delete the entry from the transaction-detail screen, gated on an explicit destructive-action confirmation, using the entry's `sha256sum` so a concurrently-edited ledger is never clobbered.

## Context

- Screen: `src/screens/transaction-detail-screen/transaction-detail-screen.tsx` — already holds `slice`/`sha256sum` from `useGetLedgerEntryContextQuery`.
- Confirmation via RN `Alert.alert` (destructive style) — no new dependency.
- Toast lifecycle pattern: `addEntries` in `src/screens/add-transaction-screen/add-transaction-next-screen.tsx`.

## Steps

1. Add a delete affordance (header action or footer button) with a localized destructive confirm dialog.
2. On confirm: `deleteLedgerEntrySourceSlice` with `sha256sum`; saving/success/error toasts; analytics `tap_transaction_delete`.
3. On success navigate back; stale-sha or server errors surface honestly (localized toast, entry untouched).

## Files

- `src/screens/transaction-detail-screen/transaction-detail-screen.tsx`
- `src/translations/en.ts`

## Acceptance criteria

- [x] Delete requires explicit confirmation; cancel leaves everything untouched.
- [x] Successful delete returns to the previous screen; the entry is gone from the ledger.
- [x] A stale `sha256sum` (entry changed elsewhere) shows an honest error and deletes nothing.
