## Objective

Show the parsed receipt as an editable proposed transaction — payee, date, description, postings (account + amount + currency) — and only on explicit confirm call `insertReceiptTransaction` with the `receiptObjectKey`. This is the anti-goal compliance point: AI proposes, the user reviews and confirms.

## Context

- Reuse the m8 editing vocabulary: `ListItem` rows, account changes via `/(app)/account-picker`, date via `DateTimePickerModal`.
- Input shape: `InsertReceiptTransactionInput` (`date`, `description`, `documentAccount`, `payee`, `postings[]` of `{ account, amountCurrency, amountNumber }`).
- Refresh parity: `AddTransactionCallback` + `router.back()` pattern from `add-transaction-next-screen.tsx`.

## Steps

1. Review screen (route under `app/(app)/`): parsed fields prefilled and editable; postings list with per-row account picker; totals visible.
2. Confirm button → saving toast → `insertReceiptTransaction` → success toast → refresh callback + back; errors surfaced honestly with the proposal intact for retry.
3. Analytics: `tap_receipt_confirm` with posting count; `receipt_insert_failed` on error.

## Acceptance criteria

- [ ] Nothing is written to the ledger before the user taps confirm; dismissing the screen writes nothing.
- [ ] Every parsed field is editable before confirm; the saved entry matches the reviewed values and appears in Journal/Home recents after refresh.
- [ ] Insert failure preserves the edited proposal for retry.

## Out of scope

- Editing after insert (that's m9's flow); attaching the image as a ledger document.
