## Objective

Wire Done to save the multi-leg transaction: build one `EntryInput` with N `PostingInput`s from the legs and persist it through the existing mutation, with the same toast lifecycle and Home-refresh behavior as quick add.

## Context

- Mutation hook: `src/screens/add-transaction-screen/hooks/use-add-entries-to-remote.ts`; the two-posting reference implementation is `addEntries` in `add-transaction-next-screen.tsx` (saving/success/error toasts, `AddTransactionCallback` refresh, `router.back()`).
- `EntryInput` (see `src/generated-graphql/schema.graphql`): `{ date, flag: "*", narration, payee, type: "Transaction", meta: {}, postings }`; each posting `amount` is a string `"<signed-amount> <CURRENCY>"`.
- Postings builder belongs in `multi-leg-utils.ts` (pure, testable — see t008).

## Steps

1. `buildEntryInput(legs, { date, payee, narration, currency })` in `multi-leg-utils.ts`, formatting cents back to `"-12.50 USD"`-style strings.
2. On Done: `tap_multi_leg_done` analytics with `legCount`, saving toast, `mutate({ variables: { entriesInput: [entry], ledgerId } })`, success/error toasts, then `AddTransactionCallback` refresh + `router.back()` on success (parity with quick add).
3. Surface mutation errors honestly (error toast, no silent failure).

## Files

- `src/screens/add-transaction-screen/multi-leg-screen.tsx`
- `src/screens/add-transaction-screen/multi-leg-utils.ts`

## Acceptance criteria

- [ ] Saving a 3+ leg transaction succeeds and it appears in Recent Transactions/Journal after the automatic Home refresh.
- [ ] The written entry is a valid balanced beancount transaction (spot-check the ledger text / web app).
- [ ] `tap_multi_leg_done` fires with the leg count; failures show the error toast.

## Out of scope

- Offline queueing; editing the saved transaction.
