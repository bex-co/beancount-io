## Objective

A native editor screen: open a ledger file from the browser, edit it in `CodeEditor` with beancount highlighting, and save through `updateLedgerFile` — with honest conflict handling and an unsaved-changes guard.

## Context

- New route `app/(app)/ledger-file.tsx` (params: `path`, optional `line`) mounting `src/screens/ledger-file-editor-screen/`.
- `getLedgerFile` returns `content` base64-encoded (`encoding === "base64"`) plus `sha`; decode on load, encode on save. `updateLedgerFile(content, ledgerId, message, path, sha)` returns the new `sha` — store it so consecutive saves work.
- A `sha` mismatch means the file changed remotely (e.g. web edit): surface it honestly — alert offering "reload latest" vs "keep editing"; never silently overwrite (trust bar in `.pm/GOAL.md`).
- Save `message` e.g. `"Update <path> from mobile"` — consistent with how edit-transaction writes describe themselves.
- Header: file name as title, Save button enabled only when dirty; back navigation with unsaved changes prompts discard/keep. `KeyboardAvoidingView` (see `edit-transaction-screen.tsx:202` for the pattern). Skeleton first-load with background color set.

## Steps

1. Route + screen scaffold; load via `useGetLedgerFileQuery`, base64-decode into `CodeEditor`.
2. Track dirty state; Save → base64-encode → `useUpdateLedgerFileMutation` → success toast/haptic + refresh `sha`; failure surfaces the server message.
3. Handle the conflict path and the unsaved-changes back-guard.

## Files

- `app/(app)/ledger-file.tsx` (new route)
